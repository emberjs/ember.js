# Consumable `ember-source-gxt` — Packaging Design

Design owner: packaging design expert (originally a read-only investigation pass)
Repo: `/Users/lifeart/Repos/ember.js` · Branch `gxt-rebase-main`
Status: **MVP implemented.** This was the design for the M-item from RFC §5.5
(blocker matrix row 1); the recommended mechanism (b) has since landed —
`scripts/build-gxt-package.mjs` assembles the git-ignored `dist-gxt-package/`,
the `emberSourceGxt` scenario (`smoke-tests/scenarios/scenarios.ts`) links it as
`ember-source`, and the Tier-1/Tier-2 gates (`gxt-consumable-test.ts`,
`gxt-boot-test.ts`) are green. The §6 step table below is retained as the

`@glimmer/component-gxt` Option-2 fallback (RFC §6).

## 0. Scope (honest, minimum-viable)

**In scope (what "consumable" must mean for the MVP):**

A consumer project can declare a dependency on a package that **is `ember-source` built with
the GXT backend** and have it resolve like any normal npm package:

- correct `exports` map over the GXT `dist/{dev,prod}/packages/*`;
- `@lifeart/gxt` declared as a **real, exact-pinned dependency** (so the bare
  `@lifeart/gxt` / `@lifeart/gxt/glimmer-compatibility` imports the GXT dist emits resolve
  from a normal `node_modules`);
- the `@glimmer/*` reactivity subpaths (`@glimmer/tracking`, `@glimmer/validator`, …) resolve
  to the **GXT shims** (which re-export the GXT reactive core), **not** the classic Glimmer VM;
- an addon-main (`lib/index.js`) whose `ember-addon` metadata (`renamed-modules`,
  `implicit-modules`) matches the **GXT** entrypoint set, and which exposes
  `absolutePaths.templateCompiler` so `ember-cli-htmlbars` (driven by Embroider's
  `classicEmberSupport()`) does not `TypeError` (the §5.5 S-item);
- the `smoke-tests/v2-app-template` scenario can be pointed at this package and `prepare()`
  (install/link) succeeds.

**Explicitly OUT of scope (the §5.5 L-item — do NOT pretend to solve it):**

- Compiling `.gjs`/`.gts`/`.hbs` to GXT reactive trees inside the standard
  `@embroider/vite` `ember()` pipeline. Embroider's template pipeline is hardwired to
  `babel-plugin-ember-template-compilation` → Glimmer **wire-format**; GXT needs its own
  `@lifeart/gxt/compiler` Vite plugin → `$_tag` trees. The two are mutually exclusive over
  the same `.gjs` (RFC §5.5 blocker #2). **The MVP supports the non-Embroider GXT Vite
  pipeline (the in-repo `vite.config.mjs` / `gxt-template-compiler-plugin.mjs` shape), not
  the Embroider strict-mode template-compiler integration.**
- The container-resolution bridge (`gxtEmberWrapperRedirect` / `ember-gxt-wrappers`) being
  shipped as a consumer-installable plugin (RFC §5.5 blocker #3, M). The package ships the
  **runtime**; the consumer-side compiler+resolution plugin is a separate item.
- FastBoot/SSR (RFC §7) and Ember Inspector (RFC §8).

This package being **consumable** is necessary-but-not-sufficient for the L-item; it unblocks
it without claiming to close it.

## 1. Investigation findings (evidence)

### 1.1 Classic `ember-source` package shape (root `package.json`, `lib/index.js`)

- `exports`: `"./*"` → `{ development: ./dist/dev/packages/*, production: ./dist/prod/packages/*,
default: ./dist/prod/packages/* }`, plus `"./types"` and `"./package.json"`.
- `main`: `lib/index.js` — the addon-main, which wraps `@embroider/addon-shim`'s
  `addonV1Shim(...)`. The `autoImportCompat.customizeMeta` hook injects a **hardcoded**
  `implicit-modules` list (classic `@glimmer/*` paths incl. `@glimmer/runtime`, `vm`,
  `wire-format`, `opcode-compiler`, …).
- `bin`: `ember-cli-gxt` → `scripts/ember-cli-gxt.mjs`.
- `files`: `build-metadata.json`, `blueprints`, `dist`, `docs/data.json`, `lib`, `types/stable`.
- `dependencies` **already include** `"@lifeart/gxt": "0.0.67"` (exact pin) and
  `@embroider/addon-shim`. (The in-repo `@ember/-internals/metal` statically imports
  `@lifeart/gxt/glimmer-compatibility` even on the classic path, which is why it is already a
  hard dependency.)
- `peerDependencies`: `"@glimmer/component": ">= 1.1.2"` — this is the RFC §6 symbol-identity
  fork surface.
- `ember-addon`: `{ version: 2, type: "addon", renamed-modules: {…classic set…} }`. The
  `renamed-modules` map is **generated by the build** (see 1.4).

### 1.2 The GXT Rollup build (`rollup.config.mjs`, `EMBER_RENDER_BACKEND=gxt`) — works, verified

Ran `EMBER_RENDER_BACKEND=gxt npx rollup --config` (exit 0). It emits the **same**
`dist/{dev,prod}/packages/*` layout as classic. Differences:

- `exposedDependencies()` swaps a subset of `@glimmer/*` IDs for the
  `packages/@ember/-internals/gxt-backend/*.ts` shims: `@glimmer/validator`, `@glimmer/manager`,
  `@glimmer/reference`, `@glimmer/destroyable`, `@glimmer/tracking`(+`/primitives/cache`),
  `ember-template-compiler`, `@ember/template-compilation`, `@ember/-internals/deprecations`,
  `@glimmer/application`, `@glimmer/utils`.
- The emitted `dist/.../@glimmer/validator/index.js` is the **shim**:
  `import '@lifeart/gxt/glimmer-compatibility'; export { … } from '../../shared-chunks/validator-*.js'; import '@lifeart/gxt';`.
- `GXT_DROPPED_ENTRIES` removes `@glimmer/{runtime,opcode-compiler,program,wire-format,encoder,
vm,util,global-context,node,owner}` from the **entry map** (they are not emitted as their own
  exposed package entries).
- The **only** bare specifiers the GXT dist imports are `@lifeart/gxt` and
  `@lifeart/gxt/glimmer-compatibility` — both are real exports of `@lifeart/gxt@0.0.67`
  (`.`, `./glimmer-compatibility`, `./compiler`, `./runtime-compiler`, `./ember-inspector`,
  `./test-utils`, `./suspense`). So a normal `node_modules` install resolves them, **provided
  `@lifeart/gxt` is a declared dependency** (it is).

### 1.3 ⚠️ CRITICAL: Rollup does NOT clean `dist/` — stale classic VM files contaminate a GXT build

After running the GXT build on top of a prior classic `dist/`, the output dir contains a
**mix**: 200 fresh GXT files (mtime 15:45) + **53 stale classic leftovers** (mtime 12:31).
The stale files are exactly the dropped VM packages still present from the previous classic build:

```
dist/dev/packages/@glimmer/{encoder,global-context,node,opcode-compiler,owner,program,
                            runtime,util,vm,wire-format}/index.js   ← STALE classic VM
dist/dev/packages/@glimmer/{validator,manager,reference,destroyable,tracking,application,
                            utils,env}/index.js                      ← fresh GXT shim
```

`@glimmer/runtime/index.js` (stale) still re-exports the **real** `renderComponent`,
`EnvironmentImpl`, `LowLevelVM`, … from classic shared-chunks. If shipped as-is, a consumer
that imports `@glimmer/runtime`/`@glimmer/validator` could get a **classic VM copy whose
`Tag`/`createTag`/`CURRENT_TAG` fork from the GXT shim's** — the exact dual-runtime hazard the
RFC §6 warns about. `dist/prod` has the identical problem (`runtime` prod = stale 12:31,
`validator` prod = fresh 15:45).

**Mandatory consequence:** the assembly script MUST `rm -rf dist/` before the GXT build.

### 1.4 The `renamed-modules` map differs between classic and GXT, and the build mutates `package.json`

The `packageMeta()` Rollup plugin (dev config only) **writes** `ember-addon.renamed-modules`
into the repo's root `package.json` at build time, derived from the emitted entrypoints. The
GXT build's map **drops** `@glimmer/{encoder,global-context,node,opcode-compiler,owner,program,
runtime,util,vm,wire-format}` and **adds** `@glimmer/application` + `@glimmer/utils`. (Observed
as a live `git diff` after the build; reverted with `git checkout package.json` for this
read-only pass.) The assembled GXT package's `renamed-modules` must therefore be the **GXT**
set, not the classic set — and the assembly must not leave the root `package.json` mutated.

### 1.5 The `ember-template-compiler` standalone entry is STILL the classic Glimmer compiler in GXT mode

`dist/{dev,prod}/packages/ember-template-compiler/index.js` is built from
`ember-template-compiler/minimal.ts` as a **direct Rollup input** (`esmInputs()`), which bypasses
the `exposedDependencies()` alias. So even in GXT mode it imports `@glimmer/compiler`
(`../shared-chunks/compiler-*.js`, `_preprocess`/`precompile`) and emits **Glimmer wire-format**.
The GXT shim (`gxt-backend/ember-template-compiler.ts`) is only substituted for _internal_ bare
`ember-template-compiler` imports. **Implication:** the published GXT package's standalone
template-compiler is NOT GXT-aware — it is the runtime precompile API that produces wire-format.
This is consistent with (and is part of) the L-item: real GXT template compilation is the Vite
plugin's job, not this entry's.

### 1.6 The S-item: `absolutePaths.templateCompiler`

`ember-cli-htmlbars@7`'s `lib/ember-addon-main.js` does:
`let ember = this.project.findAddonByName('ember-source'); return ember.absolutePaths.templateCompiler;`
(then `require.resolve` of it for `compilerPath`). The dev checkout's `lib/index.js`
(`addonV1Shim`) does **not** set `absolutePaths`, `@embroider/addon-shim` does not add it, and
`bin/build-for-publishing.js` (version-bump → `pnpm build` → `npm pack`) does **not** build a
top-level `dist/ember-template-compiler.js` AMD bundle nor patch the addon-main. So
`ember.absolutePaths` is `undefined` → `TypeError: Cannot read properties of undefined (reading
'templateCompiler')` whenever Embroider's `classicEmberSupport()` runs a classic prebuild that
reaches `ember-cli-htmlbars`. **This affects the classic dev checkout too** — it is a
pre-existing harness/packaging gap, not a GXT signal (RFC §5.5 harness note). The assembled GXT
package must close it (effort S).

### 1.7 `dist/` is git-ignored; the scenario harness

- `git check-ignore dist/...` → ignored. Building leaves no git pollution (only the
  `package.json` `renamed-modules` mutation from 1.4, which is tracked — must be reverted).
- `smoke-tests/v2-app-template/package.json` declares `"ember-source": "workspace:*"` and uses
  `@embroider/vite`'s `classicEmberSupport()` + `ember()` in `vite.config.mjs`.
- `smoke-tests/scenarios/scenarios.ts`: `v2AppScenarios = Scenarios.fromProject(() =>
Project.fromDir(<v2-app-template>, { linkDevDeps: true })).expand({ embroiderVite })`.
  `linkDevDeps: true` links every devDependency (incl. `ember-source` → the repo) into the
  prepared app.
- `scenario-tester` (via `fixturify-project@7`) exposes
  `project.linkDevDependency(name, LinkParams)` where
  `LinkParams = { baseDir, resolveName?, requestedRange? } | { target, requestedRange? } |
{ project }`. **`{ target: <abs dir of a real package> }` links that on-disk package under
  `name`** — i.e. you can link the assembled GXT package dir _as_ `ember-source` with zero
  app-code change.
- `@lifeart/gxt@0.0.67` is installed in the repo root `node_modules` (pnpm), so an assembled
  package placed inside the repo resolves `@lifeart/gxt` by walking up to the repo root.

### 1.8 `ember-addon.backend` means nothing today

`scripts/ember-cli-gxt.mjs` writes `pkg['ember-addon'].backend = 'gxt'` into the **consumer's**
`package.json` and `status` reads it back, but RFC §5.5/§6 confirm nothing else reads it. The
existence of a separate `ember-source-gxt` package **replaces** the flag's intended role: you
opt in by depending on the GXT package, not by flipping a flag the runtime ignores.

## 2. Recommended mechanism

### Recommendation: **(b) a separate, script-assembled package directory** named `ember-source-gxt`

A build script produces a self-contained package directory (proposed
`dist-gxt-package/` at the repo root, git-ignored) containing:

- a freshly **clean** GXT `dist/{dev,prod}/packages/*` (built after `rm -rf dist`),
- a derived `package.json` (name `ember-source-gxt`, GXT `renamed-modules`, `@lifeart/gxt`
  exact pin, etc. — see §3),
- the addon-main (`lib/`) with the `absolutePaths.templateCompiler` + GXT `implicit-modules` fix,
- `blueprints/`, `types/stable/`, `build-metadata.json` copied/derived from the source.

Consumed by scenarios/apps via `link:`/`file:` or scenario-tester `{ target: <dir> }`.

#### Why (b)

1. **Classic and GXT dists are mutually incompatible at the same subpaths.** `@glimmer/validator`
   et al. must resolve to the GXT shim in the GXT package and to the real VM in classic; you
   cannot host both behind one `exports` map without a per-subpath conditional fork that is
   fragile and, for `renamed-modules`, impossible to express as a condition.
2. **A clean build is mandatory (§1.3).** A dedicated, freshly-emptied output dir is the only
   reliable way to guarantee no stale classic VM file ships and forks reactive symbols.
3. **The GXT-specific metadata has an honest home.** `@lifeart/gxt` exact pin, the GXT
   `renamed-modules`/`implicit-modules`, and the `absolutePaths` addon-main are all GXT-only;
   a separate `package.json` states them plainly instead of smuggling them behind conditions.
4. **It matches RFC §5.2** ("a separate Rollup build variant … published as a distinct npm
   package") and lines up with the lockstep-release / exact-pin governance of §10.
5. **Zero-friction consumption.** scenario-tester's `{ target }` link swaps `ember-source` →
   the GXT package in place, so the acceptance app needs no source change.

#### Why NOT (a) — one package, switch via `exports` conditions or env

- **`exports` conditions cannot read env vars.** A custom condition (e.g. `"gxt"`) would force
  _every_ consumer to configure their resolver (`conditions: ['gxt']` in Vite/webpack/node) —
  unacceptable ergonomics and silently wrong if forgotten.
- Would ship **both** dists (≈ double the already-+28%-raw payload).
- `renamed-modules`, `implicit-modules`, the `@lifeart/gxt` dependency, and `absolutePaths`
  differ between backends and are **not** expressible as `exports` conditions.
- The `@glimmer/*` subpath fork (shim vs VM) under one package is exactly the dual-runtime
  symbol-identity hazard (RFC §6).

#### Why NOT (c) — "smarter" in-place publish from the same tree

Publishing two npm packages from the same working tree by re-running the build with a name
override is essentially (b) without the intermediate directory, but it loses the clean-room
guarantee and makes the two artifacts race over the shared `dist/` and the `package.json`
`renamed-modules` mutation. (c)'s only real content — _how CI publishes_ — is folded into (b)
as the publish step (§4 below). Recommend (b); treat the publish flow as (b)'s tail.

> _unless_ the addon is meant to coexist with classic `ember-source` in the same graph (it must
> not — they conflict). The assembly should keep `ember-source/` (matches the addon's own
> `renamed-modules` self-reference convention). Verify against `@embroider/addon-shim`'s
> expectations during implementation (open question; default to `ember-source/`).

### 3.1 The GXT addon-main (`lib/index.js`) override — closes the S-item

Reuse the classic `lib/index.js` (`addonV1Shim`) with two GXT-specific additions:

1. **Regenerate `autoImportCompat.customizeMeta`'s `implicit-modules`** to the GXT entry set
   (same drop/add as `renamed-modules`). Otherwise classic ember-auto-import builds reference
   non-existent `@glimmer/runtime/index.js` etc. (No effect on Embroider/vite, but correctness.)
2. **Expose `absolutePaths.templateCompiler`** on the addon instance so
   `ember-cli-htmlbars`'s `findAddonByName('ember-source').absolutePaths.templateCompiler`
   resolves. Two acceptable targets:
   - **Preferred (drop-in):** build a CJS/UMD `dist/ember-template-compiler.js` from
     `broccoli/glimmer-template-compiler.mjs` (which already re-exports `precompile` from
     `ember-template-compiler/minimal.ts`) and point `absolutePaths.templateCompiler` at it.
     Note this is the **classic wire-format** compiler (§1.5) — correct for the classic
     prebuild's needs, and it is what unblocks the harness for _both_ backends.
   - **Minimal:** point at the ESM `dist/prod/packages/ember-template-compiler/index.js`
     **iff** the consuming `ember-cli-htmlbars` version can `require` an ESM file (it generally
     cannot under CJS) — so prefer the CJS bundle.

```js
// lib/index.js (GXT) — appended to the classic addonV1Shim wrapper
module.exports = {
  ...shim,
  // … existing init()/included()/supportedBrowsers …
  absolutePaths: {
    // built by the assembly from broccoli/glimmer-template-compiler.mjs
    templateCompiler: path.join(__dirname, '..', 'dist', 'ember-template-compiler.js'),
  },
};
```

> The `absolutePaths` gap is pre-existing on classic too (§1.6). If the classic
> `ember-source` is fixed upstream in the same way, the GXT addon-main can simply inherit it.

## 4. Assembly script design

**Location:** `scripts/build-gxt-package.mjs` (node-only, no browser).
**Output:** `dist-gxt-package/` at repo root (add to `.gitignore`).
**Invocation:** `node scripts/build-gxt-package.mjs` (CI: a `pnpm build:gxt-package` script).

**Steps (idempotent):**

1. **Clean.** `rm -rf dist dist-gxt-package` (the `dist` clean is the §1.3 hazard fix).
2. **Snapshot** the root `package.json` (to restore the `renamed-modules` mutation, §1.4).
3. **Build GXT dist:** `EMBER_RENDER_BACKEND=gxt rollup --config`. This emits a clean
   `dist/{dev,prod}/packages/*` AND (via `packageMeta()`) rewrites the root `package.json`'s
   `ember-addon.renamed-modules` to the **GXT** set. Capture that map now.
4. **Build the CJS template-compiler** for `absolutePaths` (S-item): rollup/esbuild
   `broccoli/glimmer-template-compiler.mjs` → `dist/ember-template-compiler.js` (CJS/UMD).
5. **Build types** if shipping them: `node types/publish.mjs` (or copy `types/stable`).
   For MVP, copying `types/stable` is acceptable; declaration accuracy for the GXT shims is a
   later item (RFC §5.4).
6. **Assemble `dist-gxt-package/`:**
   - copy `dist/` (incl. the CJS `dist/ember-template-compiler.js`), `blueprints/`,
     `docs/data.json`, `types/stable/`, `build-metadata.json`, `scripts/ember-cli-gxt.mjs`;
   - copy `lib/` and apply the GXT addon-main patch (§3.1) — regenerate `implicit-modules`
     from the freshly-built `dist/dev/packages/**/index.js` set and add `absolutePaths`;
   - write the derived `package.json` (§3): start from the snapshot, set
     `name: 'ember-source-gxt'`, copy the captured **GXT** `renamed-modules`, ensure
     `@lifeart/gxt` exact pin, add `scripts/ember-cli-gxt.mjs` to `files`,
     set `ember-addon.backend = 'gxt'`.
7. **Restore** the root `package.json` from the snapshot (leave the working tree clean).
8. **Self-verify (seam check, fail-loud):**
   - assert `dist-gxt-package/dist/dev/packages/@glimmer/validator/index.js` contains
     `@lifeart/gxt/glimmer-compatibility` (GXT shim present);
   - assert **no** `dist-gxt-package/dist/{dev,prod}/packages/@glimmer/runtime/index.js`
     exists (no stale VM leaked — the §1.3 guard);
   - assert every bare specifier in the dist is in `{@lifeart/gxt, @lifeart/gxt/*}` and that
     each is an export of the pinned `@lifeart/gxt`;
   - `require('dist-gxt-package/lib/index.js')` and assert `absolutePaths.templateCompiler`
     exists and `require.resolve`s.

**Publish tail (the (c) concern):** CI runs steps 1–8, then `cd dist-gxt-package && npm pack`
(or `npm publish --tag preview`). Lockstep with `ember-source` and with the pinned
`@lifeart/gxt` per RFC §10 (changesets linked group / `lerna --force-publish`). A
`.github/workflows` job mirrors the existing `gxt-dual-build.yml` and uploads the tarball as an
artifact; the contract tests (RFC §10) gate it.

## 5. Acceptance test — what proves "consumable" (L-item open)

The acceptance vehicle is `smoke-tests/v2-app-template` consumed via scenario-tester. Add a
backend axis to `smoke-tests/scenarios/scenarios.ts`:

```ts
function emberSourceGxt(project: Project) {
  // swap ember-source -> the assembled GXT package, in place (app code unchanged)
  project.linkDevDependency('ember-source', {
    target: resolve(__dirname, '../../dist-gxt-package'),
  });
}
export const v2AppScenarios = Scenarios.fromProject(() =>
  Project.fromDir(dirname(require.resolve('../v2-app-template/package.json')), {
    linkDevDeps: true,
  })
).expand({ embroiderVite, emberSourceGxt }); // emberSourceGxt = GXT variant
```

### Tier 1 — the package-consumability gate (MUST pass; this IS the deliverable)

After `scenario.prepare()` of the `emberSourceGxt` variant, a node test (qunit, like
`basic-test.ts`) asserts, against the prepared app dir:

1. `prepare()`/install succeeds — `ember-source-gxt` and `@lifeart/gxt` both resolve from the
   app's `node_modules`.
2. `require.resolve('@glimmer/tracking', { paths: [appDir] })` resolves **into the GXT package's
   dist**, and reading that file shows it re-exports from `@lifeart/gxt` (GXT shim), not the
   classic VM.
3. **No** `@glimmer/runtime` standalone VM entry is resolvable from the GXT package (the §1.3
   clean-build guarantee holds end-to-end).
4. `require(<gxt pkg>/lib/index.js).absolutePaths.templateCompiler` is defined and
   `require.resolve`-able (the S-item is closed → `classicEmberSupport()` prebuild no longer
   `TypeError`s).
5. A `vite build --mode development` of the prepared app **reaches and passes** the
   `classicEmberSupport()` prebuild stage without the `absolutePaths` `TypeError`. (It may
   later fail at the GXT-template-compile stage — that failure is the documented L-item, see
   below — so this assertion is scoped to "the prebuild stage passes," not "the full build
   succeeds.")

### Tier 2 — runtime smoke (repo-internal, NOT a published-package guarantee; optional/stretch)

Boot a basic `Application` + one route + a template-only (or `@glimmer/component`) component
through the **in-repo GXT Vite pipeline** (the `vite.config.mjs` / `gxt-template-compiler-plugin.mjs`
shape, i.e. `GXT_MODE=true`, headless Playwright per the repo's existing harness) pointed at the
assembled package, and assert the route renders. This proves the **runtime** is wired correctly
when fed GXT-compiled templates. It uses the **unshipped** `gxt-template-compiler-plugin` +
`gxtEmberWrapperRedirect`, so it validates the runtime, not the consumer toolchain.

### Documented-unsupported (must be written into the package README / RFC, not silently failing)

- The standard `@embroider/vite` `ember()` plugin compiling `.gjs`/`.gts`/`.hbs` to GXT trees
  (L-item — mutually exclusive compiler contracts). A GXT app today must use the in-repo GXT
  Vite pipeline; that pipeline is not part of `ember-source-gxt`'s published surface.
- FastBoot/SSR (RFC §7); Ember Inspector (RFC §8); `{{mount}}`/engines (UNTESTED).
- Mixing `ember-source-gxt` and classic `ember-source` in one graph (symbol fork).

**What "works" precisely:** the package installs and resolves as a normal dependency; its
reactive core is GXT (not a forked VM copy); the classic-prebuild harness no longer crashes;
the GXT runtime renders a basic route when fed GXT-compiled templates by the in-repo pipeline.
**What is documented-unsupported:** end-to-end GXT app builds through the standard Embroider
template toolchain, FastBoot, Inspector, engines.

## 8. Bottom line

The GXT Rollup build already emits a correct, normal-`node_modules`-resolvable
`dist/{dev,prod}/packages/*` whose only external is the already-declared, exact-pinned
`@lifeart/gxt`. The packaging M-item is therefore **not** a build problem — it is an
**assembly + hygiene + metadata** problem:

1. build into a **clean** dir (or stale classic VM forks the reactive core — the one critical
   trap),
2. ship the **GXT** `renamed-modules`/`implicit-modules` (not classic),
3. expose `absolutePaths.templateCompiler` (the S-item, affects classic too),
4. point a scenario at it via `linkDevDependency('ember-source', { target })`.

Recommended: a `scripts/build-gxt-package.mjs` that assembles `dist-gxt-package/` (mechanism
(b)). MVP effort ≈ 4–6 engineer-days. The L-item (GXT template-compiler inside Embroider) stays
open and must be documented-unsupported, not faked.
